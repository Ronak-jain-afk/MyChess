BUG #1: FEN String Parsing - Incorrect Token Handling
--------
File: chess/uci.py
Lines: 18-26 (parse_position function)
Severity: CRITICAL (breaks FEN position setup)
Type: Logic Error / State Management

Code:
  def parse_position(self, args):
      if args[0] == "startpos":
          self.position = Position()
          moves_start = 1
      elif args[0] == "fen":
          fen = args[1]  # BUG: Only takes first token!
          self.position = Position(fen)
          moves_start = 2  # BUG: Wrong offset for finding 'moves'

Problem:
- UCI protocol sends: "position fen <placement> <side> <castle> <ep> <hmove> <fmove> moves <moves...>"
- After split(), args contains all 6+ tokens: ["fen", "piece_placement", "w/b", "KQkq", "-", "0", "1", "moves", ...]
- Code only takes args[1], which is just the piece placement
- This incomplete FEN string causes ValueError in Position.__init__()
- moves_start = 2 is wrong - should find 'moves' keyword dynamically since FEN has 6 parts

Example failure:
  Input: "position fen rnbqkbnr/pppppppp/8/8/8/8/PPPPPPPP/RNBQKBNR w KQkq - 0 1 moves e2e4"
  After split: ["fen", "rnbqkbnr/pppppppp/8/8/8/8/PPPPPPPP/RNBQKBNR", "w", "KQkq", "-", "0", "1", "moves", "e2e4"]
  Tries Position("rnbqkbnr/pppppppp/8/8/8/8/PPPPPPPP/RNBQKBNR")
  Fails: ValueError: Invalid FEN: expected 6 space-separated parts, got 1

Impact: FUNCTIONAL - Any "position fen" command will crash or be rejected

Suggested Fix:
  Find the "moves" keyword first, then join tokens before it:
  ```python
  elif args[0] == "fen":
      try:
          moves_idx = args.index("moves")
          fen = " ".join(args[1:moves_idx])
      except ValueError:
          fen = " ".join(args[1:])
          moves_idx = len(args)
      self.position = Position(fen)
      moves_start = moves_idx
  ```

BUG #3: Race Condition in search_worker Output
--------
File: chess/uci.py
Lines: 120-128 (search_worker method)
Severity: HIGH (threading safety)
Type: Race Condition / Concurrency Issue

Code:
  def search_worker(self):
      if self.infinite:
          self.best_move, self.score = search.negamax_root(self.position, self.depth)
      else:
          self.best_move, self.score = search.negamax_root(self.position, self.depth)

      if not self.stop_event.is_set():  # BUG: Not atomic with print!
          print(f"bestmove {self.move_to_uci(self.best_move)}")
          sys.stdout.flush()

Problem:
- Check and print operation is not atomic
- Race condition sequence:
  1. Thread calls is_set() -> returns False
  2. Main thread calls stop_search() -> sets stop_event
  3. Worker thread calls print() anyway
  4. Output sent after stop was requested!
- This violates UCI protocol which says output should stop immediately after "stop"

Impact: FUNCTIONAL/PROTOCOL - Can output after stop request, UCI GUIs will get confused

Suggested Fix:
  Use a lock or check stop_event again:
  ```python
  if not self.stop_event.is_set():
      self.stop_event.set()  # Prevent main thread race
      print(f"bestmove {self.move_to_uci(self.best_move)}")
      sys.stdout.flush()
  ```

BUG #7: stop Command Without Active Search Produces No Output
--------
File: chess/uci.py
Lines: 130-133 (stop_search method) and 175-179 (stop command handler)
Severity: HIGH (protocol requirement)
Type: Logic Error / State Management

Code:
  def stop_search(self):
      if self.search_thread and self.search_thread.is_alive():
          self.stop_event.set()
          self.search_thread.join()
          # No output!

  # In run(), stop command:
  elif parts[0] == "stop":
      self.stop_search()
      if self.best_move:
          print(f"bestmove {self.move_to_uci(self.best_move)}")
          sys.stdout.flush()

Problem:
- UCI specification states 'stop' command MUST produce 'bestmove <move>' response
- Current code only outputs if:
  1. search_thread is active, AND
  2. best_move is not None (was set in previous search)
- If stop is called with no active search, nothing is output
- If best_move is None, nothing is output (protocol violation)
- This leaves GUI hanging waiting for bestmove

Impact: FUNCTIONAL - GUI stalls waiting for bestmove response

Suggested Fix:
  Always output bestmove on stop:
  ```python
  elif parts[0] == "stop":
      self.stop_search()
      if self.best_move is None:
          self.best_move = self.position.generate_legal_moves()[0] if hasattr(self, 'position') else None
      if self.best_move:
          print(f"bestmove {self.move_to_uci(self.best_move)}")
      else:
          print("bestmove 0000")  # Null move if nothing else
      sys.stdout.flush()
  ```
